## Вопрос 1

Что выведет этот код? Объясните порядок выполнения блоков.

```java
class Demo {
    static { System.out.print("A "); }
    { System.out.print("B "); }
    Demo() { System.out.print("C "); }
    public static void main(String[] args) {
        System.out.print("D ");
        new Demo();
        System.out.print("E ");
        new Demo();
    }
}
```

**Предсказанный вывод:**

```
A D B C E B C
```

**Объяснение:**

1. **`static { }`** — выполняется **один раз** при загрузке класса в память → `A `
2. **`main` начинается** → `D `
3. **`new Demo()` первый раз:**
    - Сначала выполняется **инициализирующий блок** `{ }` → `B `
    - Затем **конструктор** `Demo()` → `C `
4. Продолжение `main` → `E `
5. **`new Demo()` второй раз:**
    - Статический блок **не выполняется повторно**
    - Инициализирующий блок → `B `
    - Конструктор → `C `

**Порядок выполнения:** `static → main → {instance block} → constructor`

## Вопрос 2

Класс `Child` в пакете `b`, класс `Parent` в пакете `a`. `Child extends Parent`. Какие члены `Parent` доступны в `Child`? А какие доступны в другом классе пакета `b`, который НЕ наследует `Parent`?

**Ваш ответ:**

| Модификатор | Доступно в `Child` (наследник, другой пакет) | Доступно в классе пакета `b` (не наследник) |
|-------------|----------------------------------------------|---------------------------------------------|
| `private`   | ❌ Нет | ❌ Нет |
| *(пакетный)* | ❌ Нет | ✅ Да (только если класс в том же пакете `a`) |
| `protected` | ✅ Да | ❌ Нет |
| `public`    | ✅ Да | ✅ Да |

**Ключевое правило:**
- `protected` даёт доступ наследникам **из любого пакета**, но не даёт доступ другим классам того же пакета, если они не являются наследниками.
- Пакетный доступ (без модификатора) работает **только внутри одного пакета**, независимо от наследования.

## Вопрос 3

Можно ли создать sealed-класс, у которого все наследники — record? Какое преимущество это даёт при использовании в switch?

**Ваш ответ:**

**Да, можно.** Пример:
```java
public sealed interface Shape permits Circle, Square {}
public record Circle(double r) implements Shape {}
public record Square(double side) implements Shape {}
```

**Преимущество в switch:**
Компилятор может проверить **исчерпываемость** (exhaustiveness) switch-выражения. Если перечислены все разрешённые подтипы, `default` не требуется:

```java
String describe(Shape s) {
    return switch (s) {
        case Circle c -> "Круг радиуса " + c.r();
        case Square sq -> "Квадрат со стороной " + sq.side();
        // ✅ Не нужен default: компилятор знает, что других вариантов нет
    };
}
```

Если позже добавить `record Triangle(...) implements Shape {}`, компилятор **укажет на ошибку** в switch, требуя обработать новый случай. Это повышает надёжность и защищает от пропущенных веток.

## Вопрос 4

Что выведет этот код? Объясните, что такое String Pool и как работает `intern()`.

```java
String a = "Hello";
String b = new String("Hello");
String c = b.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(b == c);
```

**Предсказанный вывод:**

```
false
true
false
```

**Объяснение:**

| Переменная | Где создана | Ссылка |
|------------|-------------|--------|
| `a` | Литерал `"Hello"` | → объект в **String Pool** |
| `b` | `new String("Hello")` | → **новый объект в heap** (не в пуле) |
| `c` | `b.intern()` | → ссылка на объект **из String Pool** (тот же, что у `a`) |

**String Pool** — специальная область памяти (метод-область/метапространство), где хранятся уникальные строковые литералы для экономии памяти.

**`intern()`** — метод, который:
1. Проверяет, есть ли строка с таким содержимым в пуле
2. Если есть — возвращает ссылку на объект из пула
3. Если нет — добавляет строку в пул и возвращает ссылку на неё

**Сравнение `==`:**
- `a == b` → `false` (разные объекты: пул vs heap)
- `a == c` → `true` (оба ссылаются на один объект в пуле)
- `b == c` → `false` (heap vs пул)

## Вопрос 5

Почему этот код не компилируется? Как исправить?

```java
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B { }
```

**Ваш ответ:**

**Причина ошибки:**
Класс `C` наследует два интерфейса с **одинаковыми сигнатурами `default`-методов**. Компилятор не может автоматически выбрать, какую реализацию использовать — это **конфликт наследования**.

**Ошибка компиляции:**
```
class C inherits unrelated defaults for hello() from types A and B
```

**Способы исправления:**

1. **Переопределить метод в классе `C`** (явный выбор):
```java
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello();  // или B.super.hello(), или своя логика
    }
}
```

2. **Предоставить собственную реализацию:**
```java
class C implements A, B {
    @Override
    public void hello() {
        System.out.println("C");
    }
}
```

3. **Комбинировать обе реализации:**
```java
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello();
        B.super.hello();
    }
}
```

## Вопрос 6

В чём разница между `StringBuilder` и `StringBuffer`? Приведите конкретный сценарий, когда нужен именно `StringBuffer`.

**Ваш ответ:**

| Критерий | `StringBuilder` | `StringBuffer` |
|----------|----------------|----------------|
| Потокобезопасность | ❌ Нет | ✅ Да (методы `synchronized`) |
| Производительность | ✅ Быстрее (нет накладных расходов на синхронизацию) | ❌ Медленнее |
| Использование | Однопоточные приложения, локальные переменные | Многопоточные приложения, общий доступ к буферу |

**Сценарий для `StringBuffer`:**
```java
// Многопоточный логгер, куда пишут несколько потоков
public class SharedLogger {
    private final StringBuffer buffer = new StringBuffer();
    
    public void log(String msg) {
        buffer.append(System.currentTimeMillis())
              .append(" ")
              .append(msg)
              .append("\n");  
    }
    
    public String getLogs() {
        return buffer.toString();
    }
}
```

## Вопрос 7

Что выведет этот код? Объясните, что происходит с переменной `x` в лямбде.

```java
int x = 10;
Runnable r = () -> System.out.println(x);
// x = 20;  // что будет, если раскомментировать?
r.run();
```

**Предсказанный вывод:**

```
10
```

**Объяснение:**

Лямбда-выражения в Java могут захватывать **только `final` или `effectively final`** локальные переменные.

- Переменная `x` инициализируется значением `10` и больше не изменяется → она **effectively final** → лямда может её захватить.
- Если раскомментировать `x = 20;`, переменная перестанет быть `effectively final` → **ошибка компиляции**:
  ```
  local variables referenced from a lambda expression must be final or effectively final
  ```

**Почему так:**
Лямбда захватывает **копию значения** (для примитивов) или **копию ссылки** (для объектов). Если бы переменная могла изменяться после захвата, это привело бы к неопределённому поведению и сложностям с многопоточностью.

## Вопрос 8

Напишите пример лямбда-выражения, которое нельзя заменить на ссылку на метод. Объясните почему.

**Ваш пример и объяснение:**

```java
// ❌ Нельзя заменить на ссылку на метод:
Function<String, String> addPrefix = s -> "ID: " + s;

// ✅ А это можно:
Function<String, String> toUpper = String::toUpperCase;
```

**Почему нельзя заменить `addPrefix`:**

Ссылка на метод (`Class::method`) может использоваться только когда лямбда **просто вызывает один метод** без дополнительной логики. В случае `s -> "ID: " + s`:

1. Выполняется **конкатенация строк** (оператор `+`), а не вызов метода
2. Литерал `"ID: "` — это внешнее значение, не являющееся методом
3. Нет метода вида `String.addPrefix(String)`, который делал бы то же самое

**Другие примеры, которые нельзя заменить:**
```java
Predicate<Integer> isEven = n -> n % 2 == 0;           // оператор %
Function<String, Integer> getLength = s -> s.length() + 1;  // арифметика после вызова метода
Supplier<String> getRandom = () -> UUID.randomUUID().toString(); // цепочка вызовов
```

## Вопрос 9

Чем `EnumMap` быстрее `HashMap` для ключей-enum? Опишите внутреннюю реализацию в одном предложении.

**Ваш ответ:**

`EnumMap` использует **битовый вектор на основе массива**, где индекс элемента равен `ordinal()` константы enum, что обеспечивает:
- **O(1)** доступ без вычисления `hashCode()` и разрешения коллизий
- **Минимальное потребление памяти** (несколько байт на элемент)
- **Быстрые пакетные операции** через битовые маски

**В одном предложении:**
> `EnumMap` хранит значения в массиве, индексированном по `ordinal()` констант enum, что исключает накладные расходы на хеширование и цепочки коллизий, характерные для `HashMap`.

**Дополнительно:**
- Для enum ≤ 64 констант используется `RegularEnumSet` (одно `long`)
- Для enum > 64 констант — `JumboEnumSet` (массив `long[]`)

## Вопрос 10

Дан `record Point(int x, int y) {}`. Что сгенерирует компилятор автоматически? Можно ли добавить в record изменяемое поле? Можно ли наследоваться от record?

**Ваш ответ:**

**Что генерирует компилятор для `record Point(int x, int y)`:**

1. ✅ **Приватные `final` поля** `x` и `y`
2. ✅ **Канонический конструктор** с параметрами `(int x, int y)`
3. ✅ **Геттеры** без префикса `get`: `x()` и `y()`
4. ✅ **`equals()` и `hashCode()`** по всем компонентам
5. ✅ **`toString()`** в формате `Point[x=1, y=2]`
6. ✅ Методы `@Override` для всех вышеперечисленных

**Можно ли добавить изменяемое поле?**
> ❌ **Нет, если это компонент записи.** Все компоненты `record` неявно `final`.  
> ✅ **Да, но только как дополнительное поле** (не в заголовке), однако это нарушает идеологию неизменяемости и отменяет автоматическую генерацию `equals`/`hashCode` для этого поля:
```java
record Point(int x, int y) {
    private transient List<String> cache; // ✅ можно, но не участвует в equals/hashCode
}
```

**Можно ли наследоваться от record?**
> ❌ **Нет.** Все `record` неявно `final` и не могут иметь наследников.  
> ❌ **Нет.** `record` не может наследоваться от другого класса (кроме неявного `java.lang.Record`).

>  `record` предназначен для **неизменяемых прозрачных носителей данных** — наследование противоречит этой цели.
